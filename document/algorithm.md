# 알고리즘 설명

이 문서는 사하구청 AI 상담사에 사용된 핵심 알고리즘들을 설명합니다.

---

## 1. RAG (Retrieval-Augmented Generation)

### RAG가 뭔가요?

LLM(대규모 언어 모델)은 학습 데이터에 없는 내용을 마치 아는 것처럼 답하는 **환각(Hallucination)** 문제가 있습니다. 예를 들어 "사하구청 시설관리과 전화번호 알려줘"라고 물으면, LLM이 학습 데이터에 없는 번호를 그럴듯하게 지어낼 수 있습니다.

RAG는 이 문제를 해결하는 방식입니다:

```
[기존 LLM 방식]
질문 → LLM이 학습 데이터에서 기억해서 답변 → 틀릴 수 있음 (환각)

[RAG 방식]
질문 → DB에서 관련 문서 검색 → 검색된 문서를 LLM에게 전달 → LLM이 문서 기반으로 답변
```

쉽게 비유하면:
- 기존 LLM = **오픈북 없이 시험 보기** (기억에 의존 → 틀릴 수 있음)
- RAG = **오픈북 시험** (자료를 보고 답변 → 정확도 향상)

### 우리 프로젝트에서의 RAG 흐름

```
1. 사용자: "쓰레기 분리수거 방법 알려줘"
2. 검색 단계: 벡터 DB에서 "쓰레기 분리수거"와 관련된 문서 5개를 찾음
3. 생성 단계: 찾은 문서들을 LLM에게 넘기고 "이 자료를 바탕으로 답변해줘"
4. LLM이 문서 내용만 사용하여 답변 생성
5. 답변 + 출처 URL을 함께 사용자에게 반환
```

### 관련 코드

- `chatbot/conversation.py` > `chat()` 메서드
  - 2단계(검색)와 3단계(생성)가 여기서 합쳐집니다
  - 시스템 프롬프트에 "참고자료에 있는 정보만 사용하세요"라는 규칙을 명시

## 2. 텍스트 임베딩 (Text Embedding)

### 임베딩이 뭔가요?

컴퓨터는 텍스트를 직접 비교할 수 없습니다. "쓰레기"와 "폐기물"이 비슷한 의미라는 걸 모릅니다. 임베딩은 텍스트를 **숫자 벡터(배열)**로 변환하여, 의미가 비슷한 텍스트끼리 가까운 위치에 놓이게 만듭니다.

```
"쓰레기 분리수거"  → [0.12, -0.45, 0.78, ..., 0.33]  (384차원 벡터)
"폐기물 재활용"    → [0.13, -0.44, 0.76, ..., 0.31]  (비슷한 벡터 → 가까움!)
"구청장 인사말"    → [0.88, 0.21, -0.56, ..., -0.12] (다른 벡터 → 멀다!)
```

### 우리가 사용하는 임베딩 모델

**sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2**

- 다국어 지원 (한국어 포함)
- 출력 차원: 384차원
- 로컬 CPU에서 실행 (API 비용 없음)
- 텍스트를 384개의 숫자로 이루어진 벡터로 변환

### 관련 코드

- `database_db/vector_store.py` > `embed_text()`, `embed_texts()`
  - HuggingFace 모델을 로컬에서 로딩하여 텍스트를 벡터로 변환

## 3. 코사인 유사도 (Cosine Similarity)

### 코사인 유사도가 뭔가요?

두 벡터가 얼마나 비슷한 방향을 가리키는지 측정하는 방법입니다. 값은 -1 ~ 1 사이이며, 1에 가까울수록 유사합니다.

```
코사인 유사도 = (A · B) / (|A| × |B|)

A · B    = 두 벡터의 내적 (각 차원끼리 곱해서 합산)
|A|, |B| = 각 벡터의 크기 (L2 norm)
```

예시:
```
질문 벡터: [1, 0, 1]
문서A 벡터: [1, 0, 1]  → 유사도 = 1.00 (완전 일치)
문서B 벡터: [1, 1, 0]  → 유사도 = 0.50 (부분 유사)
문서C 벡터: [0, 1, 0]  → 유사도 = 0.00 (무관)
```

### 우리 프로젝트에서의 활용

질문을 벡터로 변환한 뒤, DB에 저장된 모든 문서 벡터와 코사인 유사도를 계산합니다. 유사도가 높은 상위 5개 문서를 LLM에게 전달합니다.

```python
# vector_store.py에서 실제 계산하는 부분
similarity = np.dot(query_embedding, doc_embedding) / (
    np.linalg.norm(query_embedding) * np.linalg.norm(doc_embedding)
)
```

### 유사도 임계값

모든 결과를 보여주면 관련 없는 문서도 출처로 표시됩니다. 그래서 두 단계의 임계값을 적용합니다:

| 단계 | 임계값 | 역할 |
|------|--------|------|
| 검색 단계 | 0.5 이상 | 이 이하의 문서는 LLM 컨텍스트에서 아예 제외 |
| 출처 표시 | 키워드 매칭 | 질문의 키워드가 문서에 실제로 포함되어 있는지 확인 |

## 4. 하이브리드 검색 (Hybrid Retrieval)

### 왜 하이브리드인가?

벡터 검색만 사용하면 문제가 있습니다. 임베딩 모델 특성상, 한국어 행정 문서끼리는 전반적으로 유사도가 높게 나옵니다. "맨홀 수리"를 검색해도 "폭염 대비"가 높은 유사도로 나올 수 있습니다.

이를 해결하기 위해 **2단계 검색 전략**을 사용합니다:

```
[1단계 - 메타데이터 필터링]
질문: "쓰레기 분리수거 방법 알려줘"
→ 키워드 감지: "쓰레기", "분리수거" → 카테고리: "환경"
→ DB에서 카테고리가 "환경"인 문서만 추림 (탐색 범위 축소)

[2단계 - 벡터 유사도 검색]
→ 축소된 범위 내에서 벡터 유사도 계산
→ 상위 5개 결과 반환

[폴백]
→ 1단계에서 결과가 2개 미만이면, 필터 없이 전체 문서 대상 검색
```

### 메타데이터 감지 방식

질문에 포함된 키워드로 카테고리와 서비스 유형을 자동 감지합니다:

```python
# retriever.py의 키워드 매핑 예시
category_keywords = {
    "사하복지": ["복지", "지원", "수당", "돌봄", "보육", "장애", "노인"],
    "전자민원": ["민원", "신청", "발급", "증명", "신고"],
    "사하소개": ["사하구", "구청장", "조직", "위치"],
    ...
}
```

### 관련 코드

- `chatbot/retriever.py` > `detect_category()` - 키워드 기반 카테고리 감지
- `chatbot/retriever.py` > `search()` - 2단계 검색 + 폴백
- `database_db/vector_store.py` > `hybrid_search()` - 필터 + 벡터 검색 결합

## 5. 청크 분할 (Text Chunking)

### 왜 청크로 나누나요?

사하구청 홈페이지의 한 페이지는 길이가 제각각입니다. 너무 긴 텍스트는:
- 임베딩 품질이 떨어짐 (핵심 의미가 희석됨)
- LLM 컨텍스트 윈도우 초과 가능

그래서 적절한 크기로 잘라야 합니다.

### 분할 전략: RecursiveCharacterTextSplitter

```
설정값:
- chunk_size: 500자 (한 청크의 최대 길이)
- chunk_overlap: 50자 (청크 간 겹치는 부분)

분할 우선순위:
1. \n\n (문단 구분)  ← 가장 먼저 시도
2. \n   (줄바꿈)
3. ". " (문장 구분)
4. " "  (단어 구분)
5. ""   (글자 단위)  ← 최후의 수단
```

겹침(overlap)이 필요한 이유: 청크 경계에서 문맥이 잘리는 것을 방지합니다.

```
원본: "사하구청 쓰레기 분리수거는 매주 월/수/금에 실시됩니다. 재활용 수거는 화/목에 합니다."

[청크1]: "사하구청 쓰레기 분리수거는 매주 월/수/금에 실시됩니다. 재활용 수거는"
[청크2]: "재활용 수거는 화/목에 합니다."
          ↑ 50자 겹침으로 문맥 유지
```

### 관련 코드

- `processor/data_cleaner.py` > `DataCleaner.process()`
- `config.py` > `CHUNK_SIZE = 500`, `CHUNK_OVERLAP = 50`

## 6. 개인정보 필터링 (PII Filtering)

### 작동 방식

사용자 메시지가 LLM에 전달되기 **전에** 정규식(Regex)으로 개인정보 패턴을 검사합니다.

```python
감지 패턴:
1. 주민등록번호: \d{6}[-\s]?\d{7}        예) 990101-1234567
2. 전화번호:    01[016789][-\s]?\d{3,4}[-\s]?\d{4}  예) 010-1234-5678
3. 이메일:      이메일 형식 정규식         예) user@email.com
```

### 처리 흐름

```
사용자 입력 → 개인정보 감지?
  ├── YES → 경고 메시지 반환 (LLM에 전달하지 않음, DB에 저장하지 않음)
  └── NO  → 정상적으로 검색 + LLM 답변 생성
```

### 왜 필요한가?

1. **법적 의무**: 공공기관은 개인정보보호법을 준수해야 함
2. **민원인 특성**: 본인 확인 습관으로 주민번호 등을 입력할 가능성이 높음
3. **데이터 유출 방지**: LLM에 개인정보가 전달되면, 대화 로그 유출 시 2차 피해 발생
4. **Prompt Injection 방어**: 악의적 프롬프트로 이전 대화의 개인정보를 추출하려는 공격 차단

### 관련 코드

- `chatbot/conversation.py` > `PERSONAL_INFO_PATTERNS`, `_check_personal_info()`

## 7. 멀티턴 대화 (Multi-turn Conversation)

### 문맥 유지 방식

이전 대화 내용을 Supabase DB에 저장하고, 새 질문이 들어올 때 최근 10개 대화를 불러와서 LLM에게 함께 전달합니다.

```
사용자: "복지 서비스 알려줘"
AI: "사하구 복지 서비스에는 1) 노인복지 2) 아동복지 3) 장애인복지가 있습니다."
사용자: "2번 자세히 알려줘"  ← "2번"이 뭔지 이전 대화를 봐야 알 수 있음
AI: (이전 대화에서 2번이 아동복지임을 파악) "아동복지 서비스는..."
```

### 검색어 보강

최근 대화의 사용자 메시지를 현재 질문과 합쳐서 검색합니다:

```python
# conversation.py의 검색어 구성
search_query = "복지 서비스 알려줘" + " " + "2번 자세히 알려줘"
# → "복지 서비스 알려줘 2번 자세히 알려줘"로 검색하여 더 정확한 결과 획득
```

### 역질문 처리

질문이 모호하면 LLM이 선택지를 제시하며 역질문합니다. 이는 시스템 프롬프트에 규칙으로 명시되어 있습니다:

```
규칙2: 질문이 너무 넓거나 모호하면, 바로 답변하지 말고
구체적인 선택지를 제시하며 역질문하세요.
예: "복지 관련 문의를 주셨네요. 혹시 다음 중 어떤 분야가 궁금하신가요?
    1) 노인복지 2) 아동복지 3) 장애인복지 4) 기초생활수급"
```

### 관련 코드

- `chatbot/conversation.py` > `chat()` - 전체 대화 흐름
- `database_db/database.py` > `get_conversation_history()` - 이력 조회
- `chatbot/conversation.py` > `_build_history()` - LangChain 메시지 변환
